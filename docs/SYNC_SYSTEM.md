# Database Schemas & Sync Protocol

This document details the database schema and synchronization system configured for Zuvix.

## Database Schemas

### 1. `chat_sessions`
Tracks user conversations and intermediate agent logs:
- `id` (UUID, primary key)
- `title` (text)
- `created_at` (timestamp)
- `last_modified` (timestamp)
- `logs` (JSONB) - List of log messages representing thought processes and tool calls.

### 2. `skills`
Registry of installed and learned tools:
- `id` (text, primary key)
- `name` (text)
- `description` (text)
- `code` (text) - The executable TS/JS function source.
- `readme` (text) - Markdown instructions in `SKILL.md` format.
- `version` (text)
- `source` (text) - e.g. 'System', 'ClawHub', or 'Hermes-Learned'.
- `dependencies` (text[]) - Array of node package imports.
- `last_modified` (timestamp)

### 3. `dependencies`
Cloud tool dependencies and MCP configs:
- `id` (text, primary key)
- `name` (text)
- `type` (text) - e.g. 'mcp', 'package', 'env'.
- `status` (text) - 'active' or 'inactive'.
- `cloud_url` (text)
- `last_modified` (timestamp)

## Synchronization Protocol

Zuvix is designed as a local-first system. It stores records in browser **IndexedDB** caches first, allowing fully offline execution. 

### Conflict Resolution (CRDT)
To synchronize changes across browsers (Chrome, Firefox) and native platforms (Android, iOS, Windows, Linux) without central server locks:
1. **Timestamping:** Every database write appends a `last_modified` timestamp generated by the client.
2. **Replication Sync Loop:**
   - When connection is active, local records compare with remote tables.
   - If `local.last_modified > remote.last_modified`, the client updates the Supabase endpoint.
   - If `remote.last_modified > local.last_modified`, the client pulls the change and triggers an in-memory hot-reload.
3. **Conflict Merge:** Since tools and skills are stored textually, updates trigger full merges. If two edits happen concurrently, the latest timestamp takes priority (Last-Write-Wins).
